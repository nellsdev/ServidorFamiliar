# Design: Provision Debian 12 + Nextcloud + BorgBackup

## 1. Architecture Overview

Single machine, bare metal, no virtualization or containers. Every daemon runs directly on Debian 12.

```
[LAN] ← → [Android phones] [iOS] [Laptop]
     |
[nginx:80] ← PHP-FPM (3 static children)
     |
[MariaDB] ← Nextcloud (Files, Photos, Contacts, Calendar)
     |
[ext4 /var/www/nextcloud] ← data + config + apps
     |
[297GB HDD] (OS + Nextcloud + everything)
     |
[Borg] → [LUKS dm-crypt] → [ext4 /mnt/backup]
     |
[80GB HDD] (encrypted backup only)
```

**Power flow**: machine sleeps most of the day. Wake via WOL magic packet (user request) or rtcwake (03:00 backup). Idle detection → shutdown after 15 min no activity.

### Component responsibilities

| Layer | Component | Role |
|---|---|---|
| Web | nginx | Reverse proxy to PHP-FPM, static file serving, 2GB upload limit |
| App | PHP-FPM (Nextcloud) | Application logic, 3 static children, 512MB each |
| DB | MariaDB | Metadata, file cache, occ queries |
| Cache | APCu | Local opcode + user cache (no Redis) |
| Backup | Borg + LUKS | Encrypted daily snapshots to separate disk |
| Power | ethtool + rtcwake + systemd | WOL config, scheduled wake, idle shutdown |

## 2. Disk Layout

### 297GB HDD — `/dev/sda` (OS + Nextcloud)

| Partition | Size | FS | Mount | Purpose |
|---|---|---|---|---|
| `/dev/sda1` | 1 GB | ext4 | `/boot` | Kernel, bootloader |
| `/dev/sda2` | 4 GB | swap | - | Swap, swappiness=10 |
| `/dev/sda3` | rest | ext4 | `/` | OS + Nextcloud app + config + data |

No separate `/var` or `/home` — single root partition simplifies space management. The Nextcloud data directory lives under `/var/www/nextcloud/data`.

### 80GB HDD — `/dev/sdb` (Backup)

| Partition | Size | FS | Mount | Purpose |
|---|---|---|---|---|
| `/dev/sdb1` | whole disk | LUKS (ext4) | `/mnt/backup` | Borg repository |

The 80GB disk is fully encrypted with LUKS. The Borg repo resides on the unlocked ext4 filesystem at `/mnt/backup`. LUKS passphrase stored in a keyfile at `/root/luks-keyfile` (mode 0400, root-only) for unattended mount during backup timer. The passphrase is also printed and stored in a sealed envelope as offline recovery.

### Memory budget (4GB total)

| Component | Allocation |
|---|---|
| OS + services (idle) | ~500 MB |
| MariaDB (innodb_buffer_pool) | 512 MB |
| PHP-FPM (3 children × 512 MB) | 1536 MB |
| Borg (during backup window) | ~200 MB |
| **Total** | **~2.7 GB** |
| Headroom | ~1.3 GB |

Rationale: conservative innodb pool avoids swap thrashing. PHP-FPM static pool prevents runaway workers from exhausting memory. Borg runs sequentially with backup, not concurrently with heavy Nextcloud load.

## 3. Nextcloud Stack Configuration

### nginx

- Listen on port 80 only (LAN-only, TLS deferred to future change)
- Server name: `family-server.local`
- Root: `/var/www/nextcloud`
- Client max body size: 2 GB (family video uploads)
- PHP-FPM pass via `unix:/run/php/php8.2-fpm.sock`
- Standard Nextcloud nginx directives: static file optimization, security headers, rewrite rules for `.php` routing
- No HTTP/2 (low benefit for LAN, older nginx on Debian 12 stable may not include it by default)

```
server {
    listen 80;
    server_name family-server.local 192.168.1.100;
    root /var/www/nextcloud;
    client_max_body_size 2G;
    # ... standard Nextcloud nginx config
}
```

### PHP-FPM

- Pool `nextcloud`:
  - `pm = static`
  - `pm.max_children = 3`
  - `pm.max_requests = 500`
- `memory_limit = 512M`
- `upload_max_filesize = 2G`
- `post_max_size = 2G`
- `max_execution_time = 3600`
- OPcache: `opcache.memory_consumption=128`, `opcache.interned_strings_buffer=16`, `opcache.revalidate_freq=60`
- APCu: `apc.enabled=1`, `apc.shm_size=64M`

### MariaDB

```
[mysqld]
innodb_buffer_pool_size = 512M
innodb_log_file_size = 128M
innodb_flush_method = O_DIRECT
max_connections = 20
query_cache_size = 0          # removed in MariaDB 10.1.7+; 0 disables
query_cache_type = 0
tmp_table_size = 64M
max_heap_table_size = 64M
```

- Dedicated database: `nextcloud`
- Dedicated user: `nextcloud` with password stored in `config.php`
- Character set: `utf8mb4` (Nextcloud default)

### Nextcloud config.php

```php
<?php
$CONFIG = [
    'instanceid' => '<auto-generated>',
    'datadirectory' => '/var/www/nextcloud/data',
    'trusted_domains' => [
        'localhost',
        'family-server.local',
        '192.168.1.100',
    ],
    'overwriteprotocol' => 'http',
    'default_phone_region' => 'MX',
    'enable_previews' => false,
    'preview_max_x' => 256,
    'preview_max_y' => 256,
    'jpeg_quality' => 60,
    'memcache.local' => '\OC\Memcache\APCu',
    'backgroundjobs_mode' => 'cron',
    'log_type' => 'syslog',
    'log_level' => 2,
    'maintenance_window_start' => 3,
];
```

Key decisions:
- `enable_previews => false`: preview generation for images/video is CPU-heavy on Core 2 Duo with no GPU acceleration. Users view photos on their phones, not through the web UI thumbnails.
- `memcache.local => APCu`: single node, no Redis daemon overhead. APCu is sufficient for local opcode + user cache at this scale (3 users).
- `log_level => 2`: warnings only — debug and info would fill the syslog on a busy family server.
- `maintenance_window_start => 3`: align maintenance with 03:00 backup window.

### Enabled apps

Only these default apps remain enabled:

- Files (core)
- Photos (gallery view for auto-uploaded camera photos)
- Contacts (shared family address book)
- Calendar (shared family events)
- Dashboard (default landing page)
- Settings (admin)

All other bundled apps removed via `occ app:disable`.

## 4. Backup Design

### Architecture

```
systemd timer (03:00 daily)
       │
       ▼
borg-backup.service
       │
       ├── 1. cryptsetup luksOpen /dev/sdb1 backup (via keyfile)
       ├── 2. mount /dev/mapper/backup /mnt/backup
       ├── 3. mysqldump → gzip → /tmp/nextcloud-db-<date>.sql.gz
       ├── 4. borg create /mnt/backup::<hostname>-<date>
       │        /var/www/nextcloud
       │        /tmp/nextcloud-db-<date>.sql.gz
       ├── 5. rm /tmp/nextcloud-db-*.sql.gz
       ├── 6. borg prune /mnt/backup
       ├── 7. umount /mnt/backup
       └── 8. cryptsetup luksClose backup
```

### Borg repo initialization

```bash
borg init --encryption=repokey-blake2 /mnt/backup
```

`repokey-blake2`: key stored in the repo header (not a separate keyfile), BLAKE2b hash for faster hashing on CPUs without AES-NI (Core 2 Duo has no AES-NI).

### Backup paths

| Path | Type | Notes |
|---|---|---|
| `/var/www/nextcloud/` | App + config + data | Full tree including `config.php`, `data/` |
| `/tmp/nextcloud-db-*.sql.gz` | DB dump | Created before borg run, deleted after |

### Exclusions

```bash
--exclude '/var/www/nextcloud/data/*/files_trashbin'
--exclude '/var/www/nextcloud/data/*/cache'
--exclude '*.tmp'
--exclude '/var/www/nextcloud/updater'
```

Trashbin exclusion saves significant space — users can restore trashed files from Nextcloud interface within the 30-day retention, no need to back up the trash itself.

### Retention policy

```bash
borg prune --keep-daily 7 --keep-weekly 4 --keep-monthly 12 /mnt/backup
```

- 7 daily: one week of granular restore points
- 4 weekly: one month of weekly snapshots
- 12 monthly: one year of monthly snapshots

On 80GB disk with roughly 50GB of source data and Borg's deduplication, this fits comfortably. Monitor actual growth after 3 months.

### Systemd units

**`borg-backup.service`**:
```ini
[Unit]
Description=Backup Nextcloud to Borg repo
Wants=network-online.target
After=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/local/bin/borg-backup.sh
StandardOutput=journal
```

**`borg-backup.timer`**:
```ini
[Unit]
Description=Daily Borg backup at 03:00

[Timer]
OnCalendar=daily
Persistent=true

[Install]
WantedBy=timers.target
```

**`borg-prune.service`** (run as last step of the backup script rather than a separate timer — simpler to reason about).

### Restore procedure

Documented in `openspec/changes/first-change/recovery.md`.

Quick reference:

```bash
# Unlock and mount
sudo cryptsetup luksOpen /dev/sdb1 backup
sudo mount /dev/mapper/backup /mnt/backup

# List available archives
borg list /mnt/backup

# Extract latest
borg extract /mnt/backup::family-server-2026-06-16 /var/www/nextcloud
# Or extract specific paths
borg extract /mnt/backup::family-server-2026-06-16 \
  var/www/nextcloud/config/config.php

# Restore database
cd /tmp
borg extract --stdout /mnt/backup::family-server-2026-06-16 \
  tmp/nextcloud-db-2026-06-16.sql.gz | gunzip | \
  mysql -u nextcloud -p nextcloud
```

### Integrity verification

- After each backup run, `borg check --verify-data /mnt/backup` runs as the final step
- Monthly manual restore drill (one Saturday per month, restore a single file from 3 days ago to verify)

## 5. WOL & Power Management

### BIOS configuration

1. Enable Wake-on-LAN (may be labeled "Power On by PCI-E" or "Wake on LAN" in Power Management menu)
2. Enable RTC Alarm / Resume by Alarm for scheduled wake
3. Disable "Quick Boot" to ensure NIC initialization (some BIOS versions skip NIC init on fast boot, breaking WOL)

### Debian configuration

**`/etc/systemd/network/10-wol.link`**:
```ini
[Match]
MACAddress=<mac-address>

[Link]
WakeOnLan=magic
```

Fallback: `ethtool -s eth0 wol g` in a systemd oneshot service `wol-enable.service`.

Persist via udev rule as third fallback:

```
ACTION=="add", SUBSYSTEM=="net", KERNEL=="eth*", RUN+="/usr/sbin/ethtool -s $name wol g"
```

### Client-side WOL

- Android: "Wake on LAN" app or "Fing" (already commonly used for network scanning)
- Desktop: `wakeonlan` CLI package
- iOS: "Mocha WOL" or similar
- Magic packet target: broadcast to LAN broadcast address (192.168.1.255) with target MAC

### Scheduled wake (rtcwake)

Set via systemd timer `.service`:

```bash
#!/bin/bash
# Wake at 02:55 for 03:00 backup
sudo rtcwake -m no -l -t "$(date +%s -d '02:55 tomorrow')"
```

Run this as a oneshot service triggered by a timer at, say, 22:00 each night (sets the RTC alarm for next morning).

### Idle shutdown

**`/usr/local/bin/idle-shutdown.sh`**:

```bash
#!/bin/bash
# Shutdown after 15 minutes of no HTTP requests + no active SSH sessions
IDLE_FILE=/tmp/last-activity
TIMEOUT=900  # 15 minutes

# Update timestamp on each HTTP request via nginx access log (inotify-based)
# Or simpler: poll access log mtime
if [[ $(($(date +%s) - $(stat -c %Y /var/log/nginx/access.log))) -gt $TIMEOUT ]] && \
   [[ $(who | wc -l) -eq 0 ]]; then
    /usr/sbin/shutdown -h +1
fi
```

Trigger via systemd timer every 5 minutes:

```ini
[Timer]
OnCalendar=*:0/5
Persistent=false
```

Edge case: if a user is actively uploading (active HTTP connection), nginx access log is updated, so shutdown doesn't trigger. If upload stalls, access log stops updating → shutdown after 15 min + 1 min grace period.

### Auto-shutdown guard

Do NOT shutdown if:
- Backup is running (check `borg` process or `/tmp/backup-running` lockfile)
- SSH session is active
- Uptime < 30 minutes (prevents boot loop on power restore)

## 6. Provisioning Scripts

All scripts in `scripts/`, idempotent by design.

### `scripts/install-nextcloud.sh`

```bash
#!/bin/bash
set -euo pipefail

# 1. Install system packages
#    nginx, mariadb-server, php8.2-{fpm,gd,curl,xml,zip,intl,mbstring,bz2,mysql,apcu,ldap,imagick}
#
# 2. Secure MariaDB installation (mysql_secure_installation equivalent)
#    - Remove anonymous users, disallow remote root login, remove test db
#
# 3. Create Nextcloud database + user
#    mysql -e "CREATE DATABASE IF NOT EXISTS nextcloud"
#    mysql -e "CREATE USER IF NOT EXISTS 'nextcloud'@'localhost' IDENTIFIED BY '<password>'"
#    mysql -e "GRANT ALL ON nextcloud.* TO 'nextcloud'@'localhost'"
#
# 4. Download and extract latest Nextcloud tarball
#    wget https://download.nextcloud.com/server/releases/latest.tar.bz2
#    tar xjf latest.tar.bz2 -C /var/www/
#
# 5. Set permissions
#    chown -R www-data:www-data /var/www/nextcloud
#
# 6. Write nginx vhost to /etc/nginx/sites-available/nextcloud
#    Enable with symlink to sites-enabled
#
# 7. Write PHP-FPM pool config to /etc/php/8.2/fpm/pool.d/nextcloud.conf
#
# 8. Write MariaDB config to /etc/mysql/mariadb.conf.d/99-nextcloud.cnf
#
# 9. Write config.php
#
# 10. Run occ command to disable unused apps
#
# 11. Set up cron job for background jobs
#     echo "*/5 * * * * www-data /usr/bin/php -f /var/www/nextcloud/cron.php" > /etc/cron.d/nextcloud
```

### `scripts/setup-backup.sh`

```bash
#!/bin/bash
set -euo pipefail

# 1. Check if LUKS already exists on /dev/sdb1; skip if yes (idempotent)
#    cryptsetup isLuks /dev/sdb1 && exit 0
#
# 2. Create partition table + LUKS container
#    printf 'g\nn\n\n\n\nw' | fdisk /dev/sdb
#    echo "<password>" | cryptsetup luksFormat /dev/sdb1
#
# 3. Open and create ext4
#    cryptsetup luksOpen /dev/sdb1 backup
#    mkfs.ext4 /dev/mapper/backup
#
# 4. Write keyfile to /root/luks-keyfile (chmod 0400)
#
# 5. Add to /etc/crypttab: backup /dev/sdb1 /root/luks-keyfile
#    Add to /etc/fstab: /dev/mapper/backup /mnt/backup ext4 defaults,noauto 0 0
#    (noauto because we only mount during backup)
#
# 6. Install borgbackup via apt
#
# 7. Initialize Borg repo: borg init --encryption=repokey-blake2 /mnt/backup
#
# 8. Write backup script to /usr/local/bin/borg-backup.sh
#
# 9. Install systemd service + timer
```

### `scripts/setup-wol.sh`

```bash
#!/bin/bash
set -euo pipefail

# 1. Install ethtool if not present
#
# 2. Write systemd link file for WOL /etc/systemd/network/10-wol.link
#
# 3. Write rtcwake script to /usr/local/bin/set-wake.sh + systemd timer
#
# 4. Write idle-shutdown.sh to /usr/local/bin/
#
# 5. Write idle-shutdown timer
#
# 6. Configure nftables (see §7)
```

### Idempotency guarantees

Each script begins with a "check and skip" guard:
- `install-nextcloud.sh`: check if `/var/www/nextcloud/version.php` exists
- `setup-backup.sh`: check if `borg list /mnt/backup` succeeds; check if `/dev/sdb1` is already LUKS
- `setup-wol.sh`: check if `ethtool show wol eth0` outputs `wol g`

All scripts log to syslog with tag matching the script name.

## 7. Network Design

### Addressing

- Static IP: `192.168.1.100/24` (adjust to match actual LAN subnet)
- Gateway: router IP (typically `192.168.1.1`)
- DNS: router IP or `8.8.8.8` / `1.1.1.1`
- Hostname: `family-server`
- mDNS: `family-server.local` (via Avahi)

### `/etc/hosts` fallback

```
127.0.0.1   localhost
127.0.1.1   family-server.local family-server
192.168.1.100 family-server.local family-server
```

### nftables (firewall)

Table `inet filter`:

```
table inet filter {
    chain input {
        type filter hook input priority 0; policy drop;

        # Allow loopback
        iif lo accept

        # Allow established/related
        ct state established,related accept

        # SSH from LAN only
        tcp dport 22 ip saddr 192.168.1.0/24 accept

        # HTTP from LAN only
        tcp dport 80 ip saddr 192.168.1.0/24 accept

        # ICMP ping from LAN only
        ip saddr 192.168.1.0/24 icmp type echo-request accept

        # Allow outbound connections
        chain output {
            type filter hook output priority 0; policy accept
        }
    }
}
```

Rationale for default-drop input policy: the machine has no business receiving unsolicited traffic from outside the LAN. WOL magic packet is layer-2 (not blocked by nftables).

### Avahi (mDNS)

Install `avahi-daemon`, configure to advertise `_http._tcp` service for Nextcloud so clients can discover `family-server.local` without manual IP entry.

```
/etc/avahi/services/nextcloud.service:
<?xml version="1.0"?>
<!DOCTYPE service-group SYSTEM "avahi-service.dtd">
<service-group>
  <name>Nextcloud</name>
  <service>
    <type>_http._tcp</type>
    <port>80</port>
  </service>
</service-group>
```

### Router configuration

- DHCP reservation for `192.168.1.100` based on server MAC address (recommended but not critical — Debian uses static IP)
- No port forwarding (no internet exposure)
- No UPnP (disable on router if enabled)

## 8. Security Considerations

| Concern | Mitigation |
|---|---|
| LAN-only exposure | nftables default-drop policy; only SSH (22) and HTTP (80) open from 192.168.1.0/24 |
| Backup disk theft | LUKS encryption on 80GB HDD; keyfile on root partition (not on backup disk) |
| MariaDB credentials | Random password generated during install; stored in config.php (mode 640, www-data readable) |
| SSH access | Key-only auth; `PasswordAuthentication no`; `PermitRootLogin no`; only LAN IPs in sshd_config `AllowUsers` |
| Software updates | `apt update && apt upgrade` via cron weekly or manual; unattended-upgrades for security patches only |
| Web UI access | No admin account shared; each family member gets their own Nextcloud user; no external auth |
| Physical access | Case lock (if available); BIOS password (optional, can be reset via CMOS anyway) |

### SSH hardening (`/etc/ssh/sshd_config.d/99-hardening.conf`)

```
Port 22
AddressFamily inet
ListenAddress 192.168.1.100
PermitRootLogin no
PubkeyAuthentication yes
PasswordAuthentication no
ChallengeResponseAuthentication no
UsePAM no
AllowUsers nells
ClientAliveInterval 300
ClientAliveCountMax 2
MaxAuthTries 3
MaxSessions 2
```

### No-go decisions

- **No fail2ban**: only 3 LAN users with key-only SSH; HTTP is read-only or authenticated. No internet exposure means no brute-force attack surface.
- **No AppArmor/SELinux**: Debian ships with AppArmor but enabling it for nginx/PHP-FPM adds complexity. Accept risk for LAN-only service.
- **No HTTPS**: Spare the CPU cycles. Defer to future change if internet exposure is ever needed (Let's Encrypt + certbot).
- **No separate `/tmp` encryption**: Not worth the write overhead on an HDD. Nextcloud temp files are cleaned up after upload.

### Backup security

- Borg repo uses `repokey-blake2` encryption — the key is stored in the repo header, encrypted with the passphrase.
- LUKS keyfile at `/root/luks-keyfile` (mode 0400) stored on the root filesystem. If the entire machine is stolen, the keyfile is on the stolen disk too, but the separate 80GB backup HDD can be removed and stored offsite.
- **Recommendation for offsite**: after setup, copy the LUKS keyfile + Borg passphrase to a USB stick stored separately. Document this in recovery.md.

## Design Decisions Summary

| Decision | Rationale |
|---|---|
| Debian 12 over Ubuntu 22.04 | Fresher packages (PHP 8.2 natively), longer support window, lighter |
| Static PHP-FPM pool (3 children) | Prevents memory exhaustion; 3 children × 512 MB = 1.5 GB fits in 4 GB |
| No Redis | Another daemon with marginal benefit for 3 users; APCu is sufficient |
| No preview generation | Core 2 Duo cannot generate image/video previews at a usable speed |
| LUKS + repokey-blake2 | Encryption at rest for backup disk; BLAKE2b avoids AES-NI penalty |
| WOL + rtcwake + idle shutdown | Minimizes electricity; machine is needed only during active hours |
| Bash scripts over Ansible | Simpler for a one-person infra project; Ansible adds indirection with no multi-host target |
| Single root partition | Simpler to manage with limited disk space; no wasted reserved blocks on separate partitions |
| No HTTPS | CPU cost of TLS handshake on Core 2 Duo; LAN-only access; deferred to future change |
